  ---
  A. Drift-Hypothesen

  Sortiert nach Wahrscheinlichkeit, alle aus dem Code falsifizierbar:

  ┌─────┬────────────────────────────────────────────────────────────────────┬─────────────────────┬─────────────────┐
  │  #  │                             Hypothese                              │        Layer        │ Falsifizierbar? │
  ├─────┼────────────────────────────────────────────────────────────────────┼─────────────────────┼─────────────────┤
  │     │ Blueprint Gate als Schatten-Autorität: _blueprint_gate_blocked     │                     │                 │
  │ A1  │ wird in verified_plan gesetzt bevor ControlDecision vollständig    │ Orchestrator/Router │ ✅ direkt       │

  Wichtigste Erkenntnis: A1+A2 sind kaskadierend — Blueprint-Gate erzeugt unavailable, Grounding sieht "tool suggested
  but no evidence" → Fallback. Das ist kein Modellproblem, sondern ein Fehlerklassen-Mismatch zwischen zwei Subsystemen.

  ---
  B. Testebenen

  B1 — Contract Tests

  Ziel: Jeder Layer produziert den richtigen Output-Typ, unabhängig vom Modell.

  Was isoliert wird: Layer-Outputs als Datenstrukturen, kein LLM involviert.

  Inputs:  Fixture-Dict (thinking_plan / verification / tool_result)
  Outputs: ControlDecision, ExecutionResult, grounding_evidence
  Prüfung: Typ, Immutabilität, Felder-Vollständigkeit, keine Policy-Felder im ExecutionResult

  Driftformen sichtbar: A1 (Gate setzt plan-Felder), A5 (approved-Default)

  ---
  B2 — Transition Tests (Layer-Übergänge)

  Ziel: Was kommt rein, was geht raus — Kontrakt zwischen je zwei Layern.

  Thinking→Control:   thinking_plan → verification dict → ControlDecision
  Control→Executor:   ControlDecision + verified_plan → tool_args (kein Policy-Override erlaubt)
  Executor→Grounding: tool_result (status=ok/error/unavailable) → grounding_evidence entry
  Grounding→Output:   evidence_list → require_evidence bool → fallback-trigger ja/nein

  Driftformen sichtbar: A2 (unavailable→fallback), A4 (tech vs. policy status)

  ---
  B3 — Replay Tests

  Ziel: Bekannte problematische Chatverläufe reproduzierbar machen.

  Input:   user_text + chat_history JSON
  Output:  tool_sequence + final_response_type + keine verbotenen Muster

  Konkrete TRION-Fälle (→ Abschnitt E).

  ---
  B4 — State-Matrix Tests

  Ziel: Kombinatorische Zustände — welche Kombination führt zu welchem Output.

  Variablen: blueprint_available, tool_allowed, evidence_status, needs_chat_history, ...
  Output:    ob Fallback ausgelöst, ob approved-Tool trotzdem geblockt

  (→ Abschnitt F)

  ---
  B5 — Fuzz/Normalization Tests

  Ziel: Verschiedene Formulierungen desselben Intents → stabiles Routing.

  "Gaming Station installieren"
  "neuen gaming container starten"
  "steam headless aufsetzen"
  "Einen neuen Gaming-Container aus dem Blueprint gaming-station erstellen"
  → alle müssen blueprint_id=gaming-station routen, nie RÜCKFRAGE

  Driftformen sichtbar: A3, A1

  ---
  B6 — End-to-End Tests

  Ziel: Vollständiger Stack, aber mit gemocktem LLM.

  Thinking-Output: pre-fixtures (kein LLM-Call)
  Executor: echter MCP-Hub-Call (oder Mock)
  Output: echter Layer, verifizierter Output-Typ

  D. Failure Taxonomy

  Klasse: POLICY_DENY
  Definition: Control hat explizit approved=false gesetzt
  Symptom: hard_block=true, decision_class=hard_block
  Falscher Layer: —
  Test: INV-01: kein downstream block wenn approved=true
  ────────────────────────────────────────
  Klasse: TECH_UNAVAILABLE
  Definition: Tool nicht erreichbar (MCP-Hub down, Docker unreachable)
  Symptom: done_reason=unavailable, retry sinnvoll
  Falscher Layer: Executor
  Test: Status muss unavailable sein, kein Fallback-Text
  ────────────────────────────────────────
  Klasse: ROUTING_SOFT_BLOCK
  Definition: Blueprint-Score in Suggest-Zone → Rückfrage
  Symptom: _blueprint_gate_blocked=True, reason=blueprint_suggest_requires_selection
  Falscher Layer: Orchestrator/Router
  Test: Wird fälschlich als TECH_UNAVAILABLE klassifiziert → A2
  ────────────────────────────────────────
  Klasse: SCOPE_REGISTRY_MISMATCH
  Definition: Blueprint existiert in DB, nicht im Graph oder umgekehrt
  Symptom: blueprint_list zeigt X, request_container(X) scheitert
  Falscher Layer: Blueprint-Store
  Test: INV-02
  ────────────────────────────────────────
  Klasse: STALE_STATE
  Definition: _blueprint_gate_blocked aus Vorturn im verified_plan weitergereicht
  Symptom: Tool geblockt obwohl User neue Anfrage stellt
  Falscher Layer: Orchestrator
  Test: State isolation test
  ────────────────────────────────────────
  Klasse: CONTEXT_LOSS
  Definition: Follow-up-Turn hat needs_chat_history=false → Blueprint-Kontext verloren
  Symptom: Thinking kennt blueprint_id nicht, Router findet nichts
  Falscher Layer: Thinking
  Test: INV-09
  ────────────────────────────────────────
  Klasse: EVIDENCE_INSUFFICIENT
  Definition: Tool ran, pending_approval oder unavailable → kein extractable content → Fallback
  Symptom: "Ich habe keinen verifizierten Tool-Nachweis" obwohl Tool lief
  Falscher Layer: Grounding
  Test: INV-03, INV-08
  ────────────────────────────────────────
  Klasse: OUTPUT_OVERCLAIM
  Definition: Output generiert Fakten ohne Tool-Evidence (hallucination_risk=medium, kein Tool-Ergebnis)
  Symptom: Antwort behauptet Container läuft ohne ihn je gestartet zu haben
  Falscher Layer: Output
  Test: postcheck-Test
  ────────────────────────────────────────
  Klasse: FALLBACK_DRIFT
  Definition: Technischer Block → Grounding→Fallback → Output wirkt wie Policy-Deny
  Symptom: User denkt TRION will nicht, obwohl Tool nur nicht geroutet wurde
  Falscher Layer: Output
  Test: INV-06, INV-07
  ────────────────────────────────────────
  Klasse: BRIDGE_DEFAULT_APPROVAL
  Definition: Legacy Bridge-Pfad: verification = {"approved": True} ohne Control-Aufruf
  Symptom: Keine Control-Evidenz, trotzdem approved
  Falscher Layer: Bridge
  Test: INV-10

  # Immutable (frozen / Policy-Domain)
  @dataclass(frozen=True)
  class TestControlDecision:
      approved: bool
      hard_block: bool
      decision_class: str      # allow / warn / hard_block
      block_reason_code: str   # nur technisch (malicious_intent, pii, ...)
      tools_allowed: tuple     # leer = alle erlaubt
      source: str              # "control" | "skip:low_risk" | "skip:no_control"

  # Mutable (Runtime-Domain)
  @dataclass
  class TestExecutionResult:
      done_reason: str         # success / tech_fail / unavailable / timeout / skipped
      tool_statuses: list      # [{tool_name, status, reason}]
      # KEIN: approved, policy, blocked_by

  # Evidence (Guard-Domain)
  @dataclass(frozen=True)
  class TestGroundingEvidence:
      tool_name: str
      status: str              # ok / error / unavailable / partial
      has_extractable_content: bool
      # WICHTIG: unavailable ≠ error (verschiedene Fallback-Pfade!)

  # Replay-Fixture

  J. Abschlussbewertung

  Die 3 wahrscheinlichsten Drift-Ursachen

  1. A2 — Grounding Policy als Schatten-Autorität: enforce_evidence_when_tools_suggested=True + status=unavailable (aus
  Routing-Block) → missing_evidence_fallback. Das ist der direkte Ursache für "Ich habe keinen verifizierten
  Tool-Nachweis" obwohl Control approved hat. Kein Modellproblem. Direkter Code-Fix.
  2. A1 — Blueprint Gate überschreibt Tool-Execution trotz Control-Approval: _blueprint_gate_blocked wird in
  verified_plan gesetzt und blockiert request_container im Stream-Flow — dieser Check läuft nach Control, respektiert
  aber ControlDecision nicht als Autorität. Struktureller Contract-Bruch.
  3. A3 — Context-Verlust bei Follow-ups: Thinking setzt needs_chat_history=false für kurze Follow-up-Turns →
  Blueprint-ID nicht bekannt → Router findet nichts → Gate blockt. Policy-Problem im Thinking-Layer.

  Die 5 Tests mit größtem Erkenntnisgewinn

  1. RC-14 — Prüft genau die A1+A2-Kaskade: unavailable darf nicht missing_evidence_fallback auslösen
  2. RC-02 — Context-Verlust Follow-up: needs_chat_history bei "neuer Container bitte"
  3. RC-03 — pending_approval darf keinen Fallback auslösen
  4. INV-01 — approved=true + gate_blocked gleichzeitig: illegal
  5. RC-06 — Fuzz verschiedene Gaming-Formulierungen: alle müssen stabil routen

  Architekturannahmen die zuerst verifiziert werden müssen

  1. Ist ControlDecision wirklich die einzige Policy-Autorität? — Blueprint-Gate läuft nach Control und blockiert
  trotzdem. Das widerspricht der Architektur.
  2. Ist status=unavailable wirklich technisch und nicht Policy? — Aktuell wird es als technischer Status behandelt,
  aber Blueprint-Routing-Blocks erzeugen ihn. Das ist ein falscher Status-Code für einen Routing-Entscheid.
  3. Ist Grounding-Policy ein Guard oder eine Shadow-Authority? — enforce_evidence_when_tools_suggested=True macht sie
  zur Shadow-Authority die approved=true aushebelt.

  # core/drift_audit.py

  class DriftAuditHarness:
      """
      Zeichnet jeden Pipeline-Run auf. Macht ihn replaybar. Erkennt Drift automatisch.
      """

      def new_run(self) -> str:
          run_id = str(uuid.uuid4())[:8]
          self._runs[run_id] = {"events": [], "drift_alarms": []}
          return run_id

      def emit(self, run_id: str, layer: str, event_type: str, payload: dict):
          """Jeder Layer ruft das auf."""
          entry = {
              "ts": time.time(), "layer": layer,
              "event": event_type, **payload
          }
          self._runs[run_id]["events"].append(entry)
          self._check_alarms(run_id, entry)

      def _check_alarms(self, run_id: str, event: dict):
          for alarm_fn in DRIFT_ALARMS:
              if alarm_fn(self._collect_state(run_id)):
                  self._runs[run_id]["drift_alarms"].append({
                      "ts": time.time(), "trigger": alarm_fn.__name__
                  })

      def export_replay_fixture(self, run_id: str) -> dict:
          """Exportiert den Run als Replay-Testfall."""
          run = self._runs[run_id]
          return {
              "run_id": run_id,
              "events": run["events"],
              "drift_alarms": run["drift_alarms"],
              # Minimale Artefakte pro Run:
              "thinking_snapshot": self._find_event(run, "thinking_done"),
              "control_snapshot": self._find_event(run, "control_done"),
              "tool_statuses": self._find_event(run, "tool_executed"),
              "grounding_mode": self._find_event(run, "grounding_check"),
              "final_response_type": self._find_event(run, "output_done"),
          }
